# HW14 – эмбеддинги, FAISS, оценка retrieval и mini-RAG по базе знаний

## 1. Кратко: что сделано

- **База знаний:** Docker для Data Science (20 документов, после обновления +3). Выбрана из-за чёткой структуры, фактологичности и востребованности в MLOps.
- **Чанкинг:** RecursiveCharacterTextSplitter с `chunk_size=200`, `overlap=100` символов.
- **Векторизация:** `intfloat/multilingual-e5-small` (384d, поддержка русского языка).
- **Retrieval:** FAISS IndexFlatIP (косинусное сходство через скалярное произведение).
- **Оценка:** hit@3 = 100%, recall@3 = 93.75%.
- **Обновление:** добавлены документы (Docker Scout, BuildKit, Debug) – новые запросы стали находить актуальные источники.
- **Mini-RAG:** экстрактивный (без LLM) – TF-IDF + косинусное сходство для выбора лучших предложений из найденных чанков, успешно справляется с ответом на вопросы в рамках предметной области

- **Основные параметры чанкинга:** `chunk_size = 200` символов, `chunk_overlap = 100` символов.

## 4. Retrieval и индекс

### 4.1. Эмбеддинги и индекс

- **Какая embedding-модель или способ векторизации использовались:** `intfloat/multilingual-e5-small`.
- **Какая метрика сходства использовалась:** косинусное сходство (через скалярное произведение нормализованных векторов).
- **Какой индекс `FAISS` использовался:** `IndexFlatIP` (плоский индекс для точного поиска).
- **Какой `top_k` использовался по умолчанию:** 3.
- **Что показали первые примеры retrieval:** Запрос "Как уменьшить размер Docker образа?" нашёл документ №12 (Оптимизация) с score=0.945. Запрос "Что делать, если порт занят?" нашёл документ №20 (Ошибки) с score=0.853. Retrieval работает корректно.

### 4.2. Контрольные запросы и оценка retrieval

- **Сколько контрольных запросов использовалось:** 8.
- **Что считалось релевантным ответом:** документ (по `doc_id`), содержащий правильный ответ на запрос.
- **Какие метрики retrieval считались:** `hit@3`, `recall@3`.
- **Итоговый `hit@3`:** 100% (8 из 8 запросов успешны).
- **Итоговый `recall@3`:** 93.75%.
- **Дополнительные метрики (если были)** `rank_of_first_relevant` (сохранён в `retrieval_eval.csv`), `MRR@3`.
- **Какие запросы оказались самыми простыми:** "Что делать, если порт занят?", "Как войти в контейнер?" – чётко сформулированы, ключевые слова совпадают с текстом документов.
- **Какие запросы оказались самыми проблемными:** "Как сканировать образ на уязвимости?" (до обновления не было специализированного документа, retrieval находил общий документ по безопасности).

## 5. Эксперимент по параметрам retrieval и обновление базы знаний

- **Какой параметр сравнивался:** `top_k` (3 vs 2).
- **Какие два варианта сравнивались:** k=3 и k=2.
- **Как изменилось качество retrieval:** hit rate уменьшился до 87.5% (уменьшение k исключило некоторые успешные ответы).
- **Какой вариант вы выбрали как основной и почему:** k=3 – достаточен для большинства запросов.
- **Что именно было обновлено в базе знаний:** добавлены 3 новых документа: Docker Scout (сканирование уязвимостей), Docker BuildKit (ускоренная сборка), Docker Debug (отладка).
- **Как была выполнена переиндексация:** повторный чанкинг всех документов, перестроение эмбеддингов, создание нового FAISS индекса.
- **Что изменилось в retrieval после обновления базы знаний:** Запрос "Как сканировать образ на уязвимости?" стал находить новый документ №21 на первом месте (вместо старого №14).

## 7. Анализ

Выбранная база знаний оказалась удачной для учебного RAG: тема Docker имеет чёткую структуру, документы фактологичны, а вопросы естественно формулируются пользователями. Это позволило наглядно продемонстрировать все этапы пайплайна – от чанкинга до генерации ответа.

Чанкинг с параметрами 200/100 показал себя адекватно: чанки содержат 2-4 предложения, что достаточно для ответа на простой вопрос, но при этом не теряют контекст. Перекрытие в 50 символов помогло сохранить связанные фразы на границах разрыва.

Формальная оценка retrieval оказалась полезной – она объективно показала, что система работает в 100% случаев, и позволила идентифицировать проблемные запросы. Без метрик сложно было бы понять, действительно ли retrieval улучшился после обновления.

Эксперимент с top_k показал, что уменьшение k с 3 до 2 ухудшило hit rate для данной базы – все правильные документы содержаться в top-3. Это говорит о том, что retrieval c k=3 достаточно точен.

Обновление базы знаний (добавление 3 новых документов) наглядно продемонстрировало, как переиндексация влияет на результаты: запросы о новых технологиях стали находить актуальные источники на первых позициях.

Наиболее показательные ошибки mini-RAG связаны с тремя типами проблем: (1) слишком общие вопросы – retrieval теряет фокус; (2) составные вопросы – RAG отвечает только на первую часть; (3) вопросы вне базы – retrieval находит ближайший семантический аналог, что вводит пользователя в заблуждение.

Основные ограничения текущего учебного решения: (1) отсутствие LLM – ответы не переформулируются, а только вырезаются из контекста; (2) TF-IDF не понимает синонимы и контекст; (3) нет обработки составных вопросов.

## 8. Итоговый вывод

Для такой задачи (справочная база по Docker) я бы оставил текущий retrieval-конвейер как базовый: multilingual-e5-small + FAISS IndexFlatIP + k=3. Он даёт hit@3=100%, то есть полностью решает поставленную задачу.

Главный вывод про качество retrieval: оно критически зависит от качества чанкинга и релевантности embedding-модели. При хорошем retrieval даже простой экстрактивный RAG даёт адекватные ответы.

Главный вывод про связь retrieval и mini-RAG: качество итогового ответа напрямую определяется качеством retrieval – если retrieval нашёл правильный документ, ответ почти всегда корректен (даже без LLM). Ошибки retrieval напрямую ведут к ошибкам RAG.
