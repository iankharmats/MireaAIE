# Итоговый проект по курсу «Инженерия Искусственного Интеллекта»

В этой папке находится итоговый мини-проект по курсу.  
Проект должен демонстрировать применение методов и инструментов инженерии ИИ: работу с данными, модели, пайплайны, сервис, эксперименты и (по возможности) воспроизводимость.

В основе проекта лежит обработка датасета человеческих лиц, извлечение ключевых точек лица, постороение векторов преувеличения и карт признаков, дообучение ControlNet + SD 1.5 посредством RLAIF с VLLM (Qwen). ## 2. Структура проекта

Проект организован в следующей структуре:

- `requirements.txt` – зависимости проекта (библиотеки Python, необходимые для запуска).
- `report.md` – отчёт по проекту (постановка задачи, данные, эксперименты, результаты).
- `self-checklist.md` – чеклист самопроверки проекта перед сдачей.
- `notebooks/` – экспериментальные ноутбуки:
  - `feature_extracttion.ipynb` - визуализация данных и  извлечения признаков
  - `dpo_train.ipynb` - генерация датасета и DPO-обучение, подсчет метрик и сравнение с baseline
- `src/` – основной код проекта:
  - `api.py` - запуск сервиса
  - `feature_extraction.py` - модуль извлечения признаков
  - `caricature_generator.py` - модуль генерации и ранжирования карикатур 
  - `dublicates_deletion.py` - удаление дубликатов изображений из датасета `data/Humans`
- `data/` – демонстрационные/учебные данные:
  - `demo/` - демонстрационные изображения для проверки работоспособности сервиса/ноутбуков
  - `Humans/` - открытый датасет kaggle.com (> 3000 лиц анфас)
  - `dpo_dataset` - синтетический датасет изображений (сэмпл - входное изображение + по 2 варианта варпов и карикатур) для дообучения ControlNet
- `configs/` – конфигурационные файлы:
  - `.yaml`, `.json`, `.env.example` и т.п.
  - `mean_face.json` - параметры среднего лица, сформированного по данным, служат для построения векторов преувеличения
- `tests/` – тесты (юнит-тесты, простые проверки).
- `artifacts/` – сохранённые модели, отчёты, результаты экспериментов.
  - `models` - обученная модель ControlNet
  - `pretrained_models/` - предобученная модель MediaPipe
  - `generation_pipeline/` - визуализация различных этапов обработки данных (структурировано по примерам входного изображения)
  - `visualisation/` - визуализация пайплайна обработки данных

## 4. Как запустить проект

### 4.1. Запуск обучения модели

- Перейти в `project/notebooks/dpo_train.ipynb`
- Выполнить все ячейки (`Run all`)

### 4.2. Запуск сервиса FastAPI

```bash
cd project
source .venv/bin/activate # при необходимости
uv run uvicorn src.api:app --reload --port 8000
```
Сервис поднимается на порту 8000;

### Эндпоинты

#### SYSTEM
- `GET /health` - health-check сервиса.
- `GET /metrics` - статистика по работе сервиса: общее количество реквест-запросов, среднее время выполнения запросов

#### MAIN
- `POST /generate` - генерация шаржа.
- `POST /warp` - тестовый эндпоинт для проверки деформации (возвращает промежуточное изображение).
##### Пример запроса через curl:
```bash
curl -X 'POST' \
  'http://127.0.0.1:8000/generate' \
  -H 'accept: image/png' \
  -H 'Content-Type: multipart/form-data' \
  -F 'file=@face.jpg;type=image/jpeg' \
  -F 'strength=2.0' \
  --output caricature_result.png
```

## 6. Тесты

Sanity-checks (Проверка работоспособности системы)

    Тестируют системные эндпоинты /health и /metrics. Они проверяют, что сервер в принципе запускается, роутинг работает, переменные окружения доступны, а синтаксических ошибок в коде нет.

Модульные / Компонентные тесты с Mock-заглушками (Быстрые)

    Проверяют эндпоинт /warp, изолируя тяжелую функцию warp_image с помощью механизма unittest.mock. Это позволяет мгновенно проверить валидацию входных данных (например, отправку некорректного файла или слишком большой strength) без реального запуска MediaPipe.

Интеграционные тесты (E2E / Тяжелые)

    Выполняют реальный сквозной запрос к основным эндпоинтам с отправкой настоящей картинки test_image.jpg. Проверяют, что вся цепочка (чтение файла -> MediaPipe -> (SD + Qwen) -> генерация PNG ответа) отрабатывает со статусом 200 и возвращает валидный бинарный поток изображения.

Пример:

```bash
cd project
source .venv/bin/activate
python -m pytest tests
```

## 7. Демонстрация на защите

2. **Ключевые сценарии:**
- **Сценарий 1:** загружаю произвольное фото человека из директории `data/demo/Humans/` на эндпоинт `/generate` → сервис генерирует 3 варианта → Qwen выбирает лучший → показываю результат. Комментирую, на каких отличительных чертах человека алгоритм сконцентрировал внимание. Затем загружу тоже изображение на эндпоинт `/warp`, показываю деформацию лица до подачи на генеративную модель. 
- **Сценарий 2:** демонстрация метрик: отправляю запрос на `/metrics`, показываю долю артефактов и схожесть последнего вызова `/generate`

3. **На что обратить внимание преподавателя:**
- Качество итоговой карикатуры (узнаваемость лица) и отличие от обычной деформации.
- Прикачества после дообучения ControlNet и Qwen - кратко покажу `dpo_train.ipynb`.
- Архитектура: все этапы (детекция → warp → ControlNet → Qwen-критик) связаны в единый пайплайн, выборочно покажу визуализацию `artifacts/`.
- Полная воспроизводимость: проект запускается по `README.md`, все модели загружаются скриптами, секреты не закоммичены.

## 8. Ограничения и дальнейшая работа

Текущие ограничения:

  - Проект требует большой объем VRAM для запуска, все вычисления производятся локально.
  - Датасет сильно ограничен в объеме и обучение занимает очень много времени.
  - Использувается исключительно синтетический датасет для RLAIF-пайплайна обучения.
  - Веб-интерфейс не поддерживает пакетную обработку.

В дальнейшем:

  - Собрать датасет с ручной разметкой для повышения качества генерации.
  - Усложнить расчетную формулу вектора преувеличения, извлекать depth map изображения для подачи на генеративную модель.
  - Ускорить время генерации, перенести часть вычислений на удаленные сервера.
